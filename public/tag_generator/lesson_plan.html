Module 1

Web Fundamentals Crash Course

No coding — just the mental model you need before writing React

HTML & the DOM

HTML is markup — nested elements forming a tree (like XML or XAML). The browser parses it into the DOM — a live tree of objects in memory, like Unity's scene hierarchy. React generates and updates this tree for you — you rarely touch the DOM directly.

CSS & Tailwind

CSS styles DOM elements. We use Tailwind CSS, which means you apply utility class names directly in JSX instead of writing CSS files: className="flex items-center gap-4 bg-slate-800 rounded-lg p-4". Tailwind IntelliSense will autocomplete these. You'll learn them by doing.

JavaScript vs C# — key differences
C#TypeScript equivalent
string name = "x";const name = "x";
Dictionary<string, int>Record<string, number>
list.Where(x => x.hp > 0)arr.filter(x => x.hp > 0)
list.Select(x => x.name)arr.map(x => x.name)
list.FirstOrDefault(x => ...)arr.find(x => ...)
list.Aggregate(seed, fn)arr.reduce((acc, x) => fn, seed)
Task<T>Promise<T>
a ?? ba ?? b (identical!)
obj?.Propobj?.prop (identical!)

Key difference: JavaScript is single-threaded with an event loop. No threads. async/await works similarly to C# but resolution happens on the same thread via callbacks.

npm, Vite, React

npm = NuGet. Vite = build tool (like Unity's build pipeline — compiles TypeScript, bundles files, hot-reloads in dev). React = UI library. Instead of manually updating DOM elements (like the old tag generator's vanilla JS), you declare what the UI should look like given current state and React diffs + patches the DOM automatically.

TypeScript interfaces are structural (not nominal)

In C# a class must explicitly implement IHero. In TypeScript, any object that has the right shape automatically satisfies the interface — no implements keyword. This makes JSON data fit interfaces naturally.

Record<K, V> — TypeScript's typed dictionary
// C#:  Dictionary<string, float?>
// TS:  Record<string, number | null>
type TagWeights = Record<string, number | null>;

The | is a union type — the value is either a number OR null. null in this project means "this tag is not applicable."

String literal unions replace enums
// C#:  enum ItemCategory { Weapon, Vitality, Spirit }
// TS:  string literal union:
type ItemCategory = 'Weapon' | 'Vitality' | 'Spirit';

TypeScript will enforce that only one of those three strings is ever assigned to an ItemCategory variable.

Hint — BuildValues

A hero build's values field contains four maps, all of type TagWeights: ally_weight, self_weight, enemy_weight, self_score. Open abrams.json and look at the builds[0].values structure.

Module 3

React Fundamentals

Components · Props · State · Effects — the reactive mental model

No skeleton file for this module. Read the concepts and then write a small practice component.

Components are functions that return JSX
// JSX = HTML-in-JavaScript. The compiler turns it into function calls.
function HeroCard({ hero }: { hero: Hero }) {
  return (
    <div className="bg-slate-800 p-4 rounded">
      <h2>{hero.eng_name}</h2>   // {} embeds any JS expression
    </div>
  );
}

In JSX: className instead of class, onClick instead of onclick. Everything else is normal HTML.

State — the key mental shift

To make a variable change update the UI, you must use useState. Assigning to a plain variable does not re-render.

const [search, setSearch] = useState('');
// search = current value (read this)
// setSearch = setter (call this to change the value + trigger re-render)

// WRONG — does nothing visually:
search = 'new value';

// CORRECT:
setSearch('new value');

Think of it like a Unity [SerializeField] backed by an OnValueChanged event that automatically re-renders the component when called.

Rendering lists — always use .map() with a key
{heroes.map(hero => (
  <HeroCard
    key={hero.normalized_name}  // must be unique — React uses this to track items
    hero={hero}
  />
))}

HashRouter vs BrowserRouter

GitHub Pages serves only static files. A URL like /heroes/abrams hard-refreshed returns a 404 because there's no such file. HashRouter puts the path after # — the browser never sends that part to the server, so it always loads index.html and React handles the rest. URLs look like /#/heroes/abrams.

NavLink's active className — function syntax
<NavLink
  to="/heroes"
  className={({ isActive }) =>
    isActive ? 'bg-indigo-600 text-white' : 'text-slate-400 hover:text-white'
  }
>

When the URL matches /heroes, isActive is true. Add end prop to the / Calculator link so it only highlights when exactly on /, not when on /heroes.

Layout Route + Outlet

React Router's "layout route" pattern: a parent route renders a shared UI shell (the sidebar), and child routes render inside <Outlet /> — the placeholder that gets replaced with whichever child route matched.

import.meta.glob — Vite's bulk importer

This is a Vite-specific feature — the pattern is replaced at build time with code that imports all matching files. Think of it as Resources.LoadAll<HeroData>("Heroes") in Unity. The result is a plain object keyed by file path.

const heroFiles = import.meta.glob<Hero>('./heroes/*.json', { eager: true, import: 'default' });
// heroFiles = { './heroes/abrams.json': { eng_name: 'Abrams', ... }, ... }

for (const hero of Object.values(heroFiles)) {
  heroes[hero.normalized_name] = hero;
}
Zustand — reading and writing from any component
// Subscribe to a slice of the store (re-renders only when that slice changes):
const mode = useDataStore(s => s.mode);
const data = useDataStore(s => s.getData());

// Get actions (functions don't change, no re-renders):
const updateHero = useDataStore(s => s.updateHero);
updateHero(modifiedHero);
Hint — Immutable updates in Zustand

When you call set({ customData: ... }), you replace the whole object. Use spread to copy everything and only change what you need:

// Update one hero inside customData:
set({
  customData: {
    ...customData,              // copy tags, items
    heroes: {
      ...customData.heroes,    // copy all other heroes
      [hero.normalized_name]: hero  // replace just this one
    }
  }
});

// With useMemo: only recalculates when data, category, or search changes
const filtered = useMemo(() => {
  return Object.values(data.items)
    .filter(i => category === 'All' || i.category === category)
    .filter(i => i.name.toLowerCase().includes(search.toLowerCase()));
}, [data, category, search]);
Local copy pattern for editing

Load the item from the store into local state on mount. Edits modify the local copy. Save writes the whole item back to the store. This prevents half-edited data from showing elsewhere while you're editing.

const [localItem, setLocalItem] = useState<Item | null>(null);

useEffect(() => {
  const item = data.items[name ?? ''];
  if (!item) navigate('/items');
  else setLocalItem(item);
}, [name, data]);

The dot product — this is all the math

For each tag, multiply the item's weight by the build's weight for that tag, and sum them all. This is a standard vector dot product. The skeleton has every step broken into clear pseudocode comments.

function dotProduct(weights: TagWeights, scores: TagWeights): number {
  let sum = 0;
  for (const [tag, weight] of Object.entries(weights)) {
    if (weight == null) continue;   // null = skip
    sum += weight * (scores[tag] ?? 0); // missing = treat as 0
  }
  return sum;
}

TanStack Table — headless sortable table

TanStack Table handles sort state and provides row/cell iterators — you write the HTML/JSX. The skeleton has the column definition pattern and the render loop written out in comments. Key: clicking a header calls h.column.getToggleSortingHandler() which manages the sort state for you.

useDraggable vs useSortable

In the tag manager we used useSortable (items in a list reordering). The calculator uses useDraggable (dragging from a pool to a zone) + useDroppable (the drop zones). The skeleton uses useDraggable for hero cards and useDroppable for each team zone.
